Principles

01 — CLARITY OVER DENSITY

Say less, with room to breathe.

When clarity and density conflict, clarity wins.

02 — HIERARCHY IS EARNED

Emphasis through type, not noise.

Rank with size and weight before reaching for color.

03 — CALM & RESPONSIVE

Motion confirms, never decorates.

Every interaction earns a small, fast response.

04 — WARM & HONEST

Editorial warmth, plain truth.

Warm-paper, ink-on-page feel. Copy is direct.

6. When to use

One primary per view. Match the variant to the weight of the action.

Wrong — two competing primaries

Two primaries fight for attention — the user can't tell which matters more.

Right — clear hierarchy

One primary anchors the action; secondary recedes. Hierarchy is instant.

Use whenAvoid when
PrimaryThe single most important action on the view — submit, confirm, create.More than one exists on the same screen. Ever.
SecondaryA relevant alternative alongside a primary — save draft, cancel, skip.It's the only button on the view — reach for primary.
GhostLow-weight actions in toolbars, tables, or inline — edit, view, copy.Actions that need immediate clarity; ghost recedes too far.
DestructiveIrreversible or high-consequence actions — delete, revoke, remove.Any action the user can undo. Separate it spatially from primary.

9. Accessibility

Every button passes AA and remains operable without a pointer.

Touch targetMinimum 44 × 44 px (size/control-md). Small (32px) buttons need 6px clear padding around them to reach 44px.
Focus ring3px clay ring at 35% opacity, 2px offset. Never suppress it — the ring is the only keyboard affordance.
Disabled statearia-disabled="true" not the HTML disabled attribute — keeps the button focusable so keyboard users know it exists and why.
Loading statearia-busy="true" blocks activation. Screen readers announce "Saving, busy" if the visible label reads "Saving…".
Color contrastAll variants meet WCAG AA (4.5:1 text). White on ink = 15.6:1 (AAA). White on danger = 5.9:1 (AA).
Icon-onlyAlways provide an aria-label describing the action. A tooltip is a supplement — it can't replace the label.

9. Do & Don't

✓ Do
One primary per view — maximum.
Lead the label with a verb.
Sentence case, 1–3 words.
Show loading only after 300ms delay.
Pair Destructive with a confirmation step.
✗ Don't
Stack two primary buttons side-by-side.
Use disabled unless reason is obvious.
Use vague labels — "Click here", "Submit".
Mix icon-only and labeled in one group.
Use Destructive for reversible actions.

6. Labels & helper text

Every input needs a visible label. Helper text sets expectations before error text corrects them.

Label always visibleNever use placeholder text as a label — it disappears on first keypress. Always place a label above the field.
Required / optionalMark required with * after the label + add aria-required="true" to the input. The asterisk alone is not read by screen readers. Add a form-level note ("* Required") near the first field. Switch to marking (optional) only when ≥80% of fields are required.
Helper textSits below the field at text/caption size. Sets expectations, shows format examples, or explains constraints — before the user makes an error.
Error textReplaces helper text on error. Specific: "Enter a date after today" not "Invalid date". Always pair with a visual indicator — never color alone.

7. Validation

Validate inline, on blur. Never block submission without telling the user exactly what to fix.

Do
—Validate on blur (when user leaves the field)
—Show error text below the specific field
—Pair error color with text + icon
—Say what's wrong and how to fix it
Don't
—Validate on every keypress — it's exhausting
—Use a single top-of-form error banner only
—Write "Invalid input" — it tells the user nothing
—Clear a field's value after a validation error
Reward early · punish late

Show that a fix worked immediately. Wait until the user leaves the field before flagging a first-time error. Once a field has errored, switch to live re-validation so the error clears the moment it's fixed.

8. Content guidelines

Copy rules for labels, placeholders, helper and error text — one source, not repeated across sections.

PlaceholderShort hint showing expected format: "DD / MM / YYYY" or "+1 (555) 000-0000". Never a repeat of the label.
LabelsNoun phrases, not questions. "Email address" not "What is your email?" — labels scan faster in 2 words.
Length1–4 words for a label. Helper text max 1 short sentence. Error text max 1 sentence, specific action.
Sentence caseAll labels, placeholders, helper and error text in sentence case. No title case, no all-caps.

9. Keyboard & interaction

Behavior that Figma cannot capture — every build must honor this.

Tab / Shift+TabKeyboard nav: triggers :focus-visible — shows the 2px solid ink outline. Mouse/touch: triggers :focus only — shows the clay border + ring.
EscapeClears a search field (type=search) and returns focus. No effect on other types.
EnterSubmits the closest form. Inside a multi-line textarea, inserts a newline instead.
Arrow keysFor number inputs: increment/decrement by step. For date: move between day/month/year segments.
AutocompleteRespect browser autocomplete attribute — set it explicitly to prevent unintended fill.

10. Mobile & input types

The right type attribute triggers the correct mobile keyboard. Two rules prevent the most common iOS bugs.

type / inputmodeKeyboard shownUse for
type=emailEmail keyboardEmail address fields — shows @ and .com keys
type=telNumeric keypadPhone numbers — no decimal, just digits + punctuation
type=urlURL keyboardWebsite or link fields — shows / and .com keys
inputmode=decimalDecimal keypadQuantities, prices — avoids phone formatting of type=tel
type=searchSearch keyboardShows a "search" return key; clears on Esc
Two iOS rules
Auto-zoom preventionSet font-size ≥ 16px on inputs. iOS zooms the viewport when an input is smaller — our Medium (15px) is borderline; Large (17px) is safe. Never use Small (13px) for primary form inputs on mobile.
Flexible inputAccept format variations for phone, date, and currency — normalize on the backend, not in the validator. Never split one value across multiple inputs (e.g., three date fields). Use one field + backend normalization.

11. Accessibility

Every input is labelled, describable, and operable without a pointer.

Label associationhtmlFor on label + id on input — always. Never label by proximity alone.
Requiredaria-required="true" on the input in addition to the visual asterisk.
Error statearia-invalid="true" on the input + aria-describedby pointing to the error message element.
Helper textaria-describedby pointing to the helper text element — screen readers announce it after the field name.
Focus ring:focus (click/tap) — clay border 2px + clay ring 3px. :focus-visible (Tab) — outline: 2px solid ink, offset 2px. Use outline not box-shadow — it survives Windows High Contrast / forced-colors mode. Never suppress either state.
DisabledUse aria-disabled="true" + tabindex="-1" when the field is truly inert. Otherwise prefer read-only which keeps the value copyable.

6. Do & Don't

✓ Do
One primary action per card.
Use the simplest pattern that fits.
Keep headlines to 1–2 lines max.
Separate by gap, not nested cards.
Flat at rest — elevation on hover only.
✗ Don't
Stack a heavy shadow at rest.
Nest cards more than one level.
Use a left-border accent stripe.
Crowd three ideas into one card.
Apply hover styles to static cards.

7. Accessibility

Tappable cardWrap in <a> or <button>. Never attach onClick to a bare <div>.
Focus ringVisible focus ring on interactive cards — 3px clay offset ring, same as buttons.
Selectablearia-selected="true/false" + role="option" or "checkbox".
ImagesAlways provide alt. For decorative images use alt="".
ContrastBody text (#16243A on white) 15.5:1 AAA. Dark surface body (white/50% on ink) passes AA — use white/70%+ for body copy.
